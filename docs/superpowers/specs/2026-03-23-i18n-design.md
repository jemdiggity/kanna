# i18n Design Spec

## Overview

Add internationalization to Kanna's Vue frontend, supporting English, Japanese, and Korean. All user-facing strings are currently hardcoded in English across 23 Vue components and several composables. Rust backend has no user-facing strings — i18n is entirely a frontend concern.

## Library

**vue-i18n** in Composition API mode (`legacy: false`). Chosen for its Vue 3 ecosystem support, `useI18n()` composable, and reactive locale switching. No build-time compilation plugin — overkill for ~200 static strings.

## File Structure

```
apps/desktop/src/
  i18n/
    index.ts              # createI18n setup, export plugin
    locales/
      en.json             # English (source of truth)
      ja.json             # Japanese
      ko.json             # Korean
```

## Locale File Format

Keys grouped by domain, flat within each domain. Example `en.json`:

```json
{
  "tasks": {
    "newTask": "New Task",
    "makePR": "Make PR",
    "closeTask": "Close / Reject",
    "descriptionPlaceholder": "Describe the task..."
  },
  "actions": {
    "create": "Create",
    "cancel": "Cancel",
    "done": "Done",
    "delete": "Delete",
    "close": "Close",
    "edit": "Edit",
    "submit": "Submit"
  },
  "preferences": {
    "title": "Preferences",
    "language": "Language",
    "suspendAfter": "Suspend After (min)",
    "killAfter": "Kill After (min)",
    "ideCommand": "IDE Command"
  },
  "shortcuts": {
    "title": "Keyboard Shortcuts",
    "showOnStartup": "Don't show on startup",
    "groupTasks": "Tasks",
    "groupViews": "Views"
  },
  "sidebar": {},
  "modals": {},
  "toasts": {}
}
```

`ja.json` and `ko.json` have identical key structure with translated values. Missing keys fall back to English automatically (vue-i18n default).

## Plugin Setup

`apps/desktop/src/i18n/index.ts`:

```typescript
import { createI18n } from 'vue-i18n'
import en from './locales/en.json'
import ja from './locales/ja.json'
import ko from './locales/ko.json'

const i18n = createI18n({
  legacy: false,
  locale: 'en',
  fallbackLocale: 'en',
  messages: { en, ja, ko }
})

export default i18n
```

Registered in `main.ts` via `app.use(i18n)` before `app.mount()`.

All three locale files eagerly imported — they're small enough that lazy loading adds unnecessary complexity.

## Locale Persistence

Stored in the existing SQLite `settings` table as `key: 'locale'`, `value: 'en' | 'ja' | 'ko'`.

- Default setting `{ key: 'locale', value: 'en' }` added to `runMigrations` in `App.vue`.
- On startup (after migrations), read with `getSetting(db, 'locale')` and set `i18n.global.locale.value`.
- On user change, call `setSetting(db, 'locale', newValue)` and update `i18n.global.locale.value`.
- No app restart required — vue-i18n reactively updates all `$t()` / `t()` calls.

## Preferences UI

Add a language dropdown to `PreferencesPanel.vue`:

```html
<label>{{ $t('preferences.language') }}</label>
<select v-model="locale">
  <option value="en">English</option>
  <option value="ja">日本語</option>
  <option value="ko">한국어</option>
</select>
```

Language names displayed in their own language (standard practice — users can find their language regardless of current locale).

## Component Migration

### Templates

```html
<!-- Before -->
<h2>New Task</h2>
<button>Cancel</button>

<!-- After -->
<h2>{{ $t('tasks.newTask') }}</h2>
<button>{{ $t('actions.cancel') }}</button>
```

### Composables

Composables use `useI18n()`:

```typescript
const { t } = useI18n()
// In computed or reactive context:
{ action: "newTask", label: t('tasks.newTask'), group: t('shortcuts.groupTasks') }
```

Since shortcut definitions are in `computed()`, they reactively update on locale change.

### Migration Order

1. `i18n/index.ts` + locale files + `main.ts` registration — infrastructure
2. `useKeyboardShortcuts.ts` / `useShortcutContext.ts` — centralized labels reused across components
3. `Sidebar.vue`, `ActionBar.vue` — always visible
4. `PreferencesPanel.vue` — add language selector + migrate existing strings
5. All modals — NewTaskModal, KeyboardShortcutsModal, AddRepoModal, etc.
6. Remaining components — TaskHeader, TagBadges, ToastContainer, etc.

## Scope

- Static strings only — no interpolation, pluralization, or date/number formatting.
- Frontend only — no Rust changes needed.
- No lazy loading — all locales bundled.
- No RTL support needed (en, ja, ko are all LTR).
